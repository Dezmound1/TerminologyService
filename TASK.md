## 1. Контекст задачи

### 1.1. Зачем нужен сервис

Существуют системы, которые обмениваются между собой электронными документами в формате JSON. Чтобы данные в полях документов трактовались всеми участниками обмена однозначно, нужно зафиксировать соглашения о кодировании контекста.

Для этого существует **независимый сервис терминологии** - база справочников. Каждый справочник содержит набор кодов и их значений. Клиентские системы используют коды из справочников в своих документах, а сервис позволяет:
- получать перечень доступных справочников,
- получать элементы конкретной версии справочника,
- проверять, существует ли элемент с заданным кодом и значением в указанной версии.

### 1.2. Пример из медицинской предметной области

**Справочник «Специальности медицинских работников»:**

| Код | Значение            |
| --- | ------------------- |
| 1   | Врач-терапевт       |
| 2   | Травматолог-ортопед |
| 3   | Хирург              |

**Справочник МКБ-10 (фрагмент):**

| Код  | Значение                                                          |
| ---- | ----------------------------------------------------------------- |
| S99  | Другие и неуточнённые травмы голеностопного сустава и стопы       |

**Пример документа «Направление к врачу»:**

```json
{
  "direction": {
    "patient":     { "full_name": "..." },
    "doctor_from": { "full_name": "...", "spec_code": 1 },
    "doctor_to":   { "full_name": "...", "spec_code": 2 },
    "diagnosis":   { "dia_code": "S99" }
  }
}
```

В документе используются коды `1`, `2` (из справочника специальностей) и `S99` (из МКБ-10). Сервис терминологии позволяет принимающей системе валидировать эти коды.

## 2. Доменные сущности

Сервис оперирует тремя сущностями, связанными иерархически: **Справочник → Версия справочника → Элемент справочника**.

### 2.1. Справочник (RefBook)

| Атрибут        | Тип             | Обязательность | Ограничения       |
| -------------- | --------------- | -------------- | ----------------- |
| Идентификатор  | целое число     | авто (PK)      | -                 |
| Код            | строка          | обязателен     | до 100 символов   |
| Наименование   | строка          | обязателен     | до 300 символов   |
| Описание       | текст           | необязателен   | произвольной длины|

### 2.2. Версия справочника (Version)

| Атрибут                       | Тип       | Обязательность | Ограничения      |
| ----------------------------- | --------- | -------------- | ---------------- |
| Идентификатор                 | целое     | авто (PK)      | -                |
| Идентификатор справочника     | FK        | обязателен     | → RefBook        |
| Версия                        | строка    | обязателен     | до 50 символов   |
| Дата начала действия версии   | дата      | обязателен     | -                |

### 2.3. Элемент справочника (Element)

| Атрибут                       | Тип       | Обязательность | Ограничения      |
| ----------------------------- | --------- | -------------- | ---------------- |
| Идентификатор                 | целое     | авто (PK)      | -                |
| Идентификатор версии          | FK        | обязателен     | → Version        |
| Код элемента                  | строка    | обязателен     | до 100 символов  |
| Значение элемента             | строка    | обязателен     | до 300 символов  |

### 2.4. Бизнес-ограничения целостности

1. **Уникальный код справочника.** Не может существовать более одного `RefBook` с одинаковым значением в поле «Код».
2. **Уникальная версия в рамках справочника.** Не может существовать более одной `Version` с одинаковой парой («Идентификатор справочника», «Версия»).
3. **Уникальная дата начала в рамках справочника.** У одного `RefBook` не может быть более одной `Version` с одинаковой датой начала действия.
4. **Уникальный код элемента в рамках версии.** В одной `Version` не может существовать более одного `Element` с одинаковым кодом.

## 3. Понятие «текущая версия»

> Это ключевое бизнес-правило. Используется в нескольких API-эндпоинтах и в админке.

**Текущая версия справочника** - это та его версия, у которой:
- дата начала действия позже всех остальных версий этого справочника,
- но не позже текущей даты.

То есть среди всех версий справочника берём те, у которых `дата_начала ≤ сегодня`, и из них выбираем версию с **максимальной** датой начала.

**Если ни одна версия не удовлетворяет условию** (например, все версии в будущем или версий нет вообще) - у справочника **нет текущей версии**. В этом случае API-методы возвращают пустой результат (см. поведение в разделе 5).

### 3.1. Пример

Справочник «Специальности медработников» имеет версии:
- v1.0, дата начала 2022-01-01
- v2.0, дата начала 2023-01-01
- v3.0, дата начала 2026-01-01

Если сегодня **2024-06-15**, текущей версией является **v2.0** (её дата начала - самая поздняя из тех, что не позже сегодняшней).

Если сегодня **2021-12-31**, текущей версии **нет** - все три версии в будущем относительно сегодняшней даты.

## 4. Административная панель

Должна быть доступна административная панель со следующими возможностями.

### 4.1. Управление справочниками

- Добавление, изменение, удаление справочников.
- На странице редактирования справочника заполняются: **Код**, **Наименование**, **Описание**.
- На той же странице должен отображаться список имеющихся версий этого справочника с возможностью перехода к их редактированию.
- В списке справочников отображаются колонки: **Идентификатор**, **Код**, **Наименование**, **Текущая версия**, **Дата начала действия версии**.

### 4.2. Управление версиями справочников

- Добавление, изменение, удаление версий справочников.
- На странице редактирования версии заполняются: **Справочник** (выбор из списка), **Версия** (строка), **Дата начала действия** (выбор из календаря).
- На той же странице должна быть возможность редактировать элементы этой версии.
- В списке версий отображаются колонки: **Код справочника**, **Наименование справочника**, **Версия**, **Дата начала версии**.

### 4.3. Управление элементами

- Добавление, изменение, удаление элементов в версии справочника.
- На странице редактирования элемента заполняются: **Версия** (выбор из списка вида «справочник-версия»), **Код элемента** (строка), **Значение элемента** (строка).

### 4.4. Локализация

Интерфейс административной панели должен быть **на русском языке**.

## 5. Требования к API

API должно предоставлять три GET-метода. Все эндпоинты возвращают JSON.

### 5.1. Получение списка справочников

**Маршрут:** `GET /refbooks/[?date=<date>]`

**Параметры запроса:**

| Параметр | Описание                                                                                                                                                         |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `date`   | Необязательный. Дата в формате `ГГГГ-ММ-ДД`. Если указана - возвращаются только те справочники, у которых есть версия с датой начала действия не позднее `date`. |

**Формат ответа:**

| Поле              | Тип    | Описание                |
| ----------------- | ------ | ----------------------- |
| `refbooks`        | список | Список справочников     |
| `refbooks[].id`   | строка | Идентификатор           |
| `refbooks[].code` | строка | Код справочника         |
| `refbooks[].name` | строка | Наименование справочника|

**Пример запроса:**

```
GET /refbooks/?date=2022-10-01
```

**Пример ответа:**

```json
{
  "refbooks": [
    {
      "id": "1",
      "code": "MS1",
      "name": "Специальности медработников"
    },
    {
      "id": "2",
      "code": "ICD-10",
      "name": "МКБ-10"
    }
  ]
}
```

### 5.2. Получение элементов справочника

**Маршрут:** `GET /refbooks/<id>/elements [?version=<version>]`

**Параметры запроса:**

| Параметр   | Описание                                                                                                                                                                                  |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`       | Идентификатор справочника (в URL).                                                                                                                                                        |
| `version`  | Необязательный. Версия справочника. Если не указана - возвращаются элементы **текущей версии** (определение в разделе 3). |

**Формат ответа:**

| Поле               | Тип    | Описание                            |
| ------------------ | ------ | ----------------------------------- |
| `elements`         | список | Список элементов в версии справочника |
| `elements[].code`  | строка | Код элемента                        |
| `elements[].value` | строка | Значение элемента                   |

**Пример запроса:**

```
GET /refbooks/1/elements?version=1.0
```

**Пример ответа:**

```json
{
  "elements": [
    {
      "code": "J00",
      "value": "Острый назофарингит"
    },
    {
      "code": "J01",
      "value": "Острый синусит"
    }
  ]
}
```

### 5.3. Валидация элемента

**Маршрут:** `GET /refbooks/<id>/check_element?code=<code>&value=<value>[&version=<version>]`

**Назначение:** проверка того, что элемент с заданным кодом и значением присутствует в указанной версии справочника.

**Параметры запроса:**

| Параметр  | Описание                                                                                                                                                                                  |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`      | Идентификатор справочника (в URL).                                                                                                                                                        |
| `code`    | Обязательный. Код элемента.                                                                                                                                                               |
| `value`   | Обязательный. Значение элемента.                                                                                                                                                          |
| `version` | Необязательный. Версия справочника. Если не указана - проверка ведётся в **текущей версии** (определение в разделе 3). |

## 7. Замечания и интерпретации

В исходном ТЗ есть места, требующие интерпретации. Решения зафиксированы здесь, чтобы и реализация, и проверяющий видели одну и ту же трактовку.

### 7.1. Формат ответа `/check_element`

В исходном ТЗ формат ответа для метода валидации **не определён**. Решил возарщать так:

- HTTP-статус: `200 OK` (даже если элемент не найден - это валидный ответ, не ошибка).
- Тело ответа: `{"exists": true}` или `{"exists": false}`.

Альтернативный вариант (`200`/`404`) отвергнут, так как 404 семантически означает «ресурса нет», а здесь ресурс (метод) есть, просто результат проверки - отрицательный.

### 7.2. Поведение, когда у справочника нет текущей версии

Если для справочника невозможно определить текущую версию (все версии в будущем относительно сегодняшней даты, или версий нет вообще), а в запросе `version` не указан:

- `GET /refbooks/<id>/elements` → `200 OK`, `{"elements": []}`.
- `GET /refbooks/<id>/check_element?...` → `200 OK`, `{"exists": false}`.

### 7.3. Коды ошибок API

| Ситуация                                                            | HTTP-статус | Тело ответа                          |
| ------------------------------------------------------------------- | ----------- | ------------------------------------ |
| Невалидный формат `date` в `/refbooks/`                             | `400`       | `{"detail": "..."}`                  |
| Не указаны обязательные `code` или `value` в `/check_element`       | `400`       | `{"detail": "..."}`                  |
| Справочник с указанным `id` не существует                           | `404`       | `{"detail": "Справочник не найден"}` |
| Справочник существует, но указанная `version` у него отсутствует    | `404`       | `{"detail": "Версия не найдена"}`    |

### 7.4. Идентификатор в ответах API

В ТЗ тип поля `refbooks[].id` указан как `string`, хотя в самой модели это целочисленный PK. Сериализатор может возвращать как `1` (число), так и `"1"` (строку) - оба варианта совместимы с `string`-полем в большинстве клиентов. **Реализуем как число** - это естественно для PK; при необходимости поведение легко изменить.
